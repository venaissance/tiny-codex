Pros

  • 完全崩溃隔离
  • 概念简单（每线程=一个进程）

Cons

  • 内存高：N 线程 × ~50MB = 很快几百MB

  • IPC 序列化开销（每条消息都要 JSON 往返）

Pros

  • 内存极低：每线程仅消息历史 ~2MB
  • API Client / 连接池共享
  • Prompt Cache 可跨线程复用
  • 线程切换零开销（同一事件循环）
  • 新线程即时启动
  • Claude Code 已验证此模式可行

Cons

  • Agent Loop 里的 bug 可能影响所有线程
  • 需要 AsyncLocalStorage 做上下文隔离
  • CPU 密集操作（如大文件 diff）需额外处理

为什么方案 D 更好？

对比维度 方案 B（多进程） 方案 D（单进程）
10 个线程内存 ~500MB ~80MB（共享+消息）
新线程启动 ~500ms（fork+init） ~1ms（new object）
API 连接复用 不可能 天然共享
Bash 崩溃隔离 完全隔离 完全隔离（spawn）
Agent Loop 崩溃 仅影响单线程 需 try/catch 保护
实现复杂度 中等（IPC 协议） 低（直接函数调用）

关键论点

Agent Loop 99% 的时间在等待 LLM API 返回（几秒到十几秒），这期间 CPU 完全空闲。 10 个线程同时跑，Node.js 事件循环毫无压力。

真正危险的操作（bash 命令）无论哪种方案都是 spawn() 子进程，隔离程度一样。

方案 B 的进程隔离解决的是一个不存在的问题（Agent Loop 的 CPU 竞争）， 却引入了真实的代价（内存、IPC 复杂度、无法共享资源）。
